# Деплой на VPS (Docker Compose) — пошагово

Инструкция рассчитана на новичка. Поднимаем тестовую версию сайта на VPS
(например, Timeweb Cloud), чтобы показать заказчику. Управляете вы с Windows.

Что получится в итоге: сайт по адресу `http://<IP-сервера>:3000`, админка —
`http://<IP-сервера>:3000/admin/login`.

Стек, который поднимется автоматически: Next.js (сайт) + PostgreSQL (база) +
том для загруженных фото. Всё в Docker, одной командой.

Есть два пути:
- **Вариант А (рекомендуется) — автоматически через cloud-init.** Сервер сам
  всё установит и запустит при первой загрузке. Почти ничего руками. См. ниже.
- **Вариант Б — вручную по шагам** (разделы 0–8). Подходит, если хотите
  понимать каждый шаг или настраиваете уже созданный сервер.

---

## Вариант А. Автоматическая установка (cloud-init)

Timeweb умеет выполнять ваш скрипт при создании сервера
([документация](https://timeweb.cloud/docs/cloud-servers/manage-servers/cloud-init)).
Мы используем готовый скрипт `deploy/cloud-init.sh` из репозитория — он сам
поставит Docker, скачает код, сгенерирует секреты и запустит сайт.

1. Откройте файл **`deploy/cloud-init.sh`** в репозитории.
2. Вверху в блоке «НАСТРОЙКИ» поменяйте как минимум:
   - `ADMIN_PASSWORD` — ваш пароль для входа в админку;
   - при необходимости `REPO_URL` (если у вас свой форк).
3. При **создании сервера** в Timeweb:
   - ОС: **Ubuntu 24.04**;
   - найдите поле **«Cloud-init / скрипт инициализации»** и вставьте туда
     **всё содержимое** отредактированного `deploy/cloud-init.sh`;
   - создайте сервер.
4. Подождите 5–10 минут (идёт установка и сборка). Затем откройте
   `http://<IP-сервера>:3000`. Админка — `…/admin/login`, пароль — тот, что
   вы вписали в `ADMIN_PASSWORD`.

Проверить ход установки (по SSH, см. шаг 1): `cat /var/log/cloud-init-output.log`

> Репозиторий должен быть **публичным** (иначе `git clone` в скрипте не пройдёт).
> Если он приватный — сделайте публичным на время теста, либо вставьте в
> `REPO_URL` ссылку с токеном вида
> `https://<TOKEN>@github.com/<аккаунт>/zodchie-dn.git`.

> Скрипт выполняется один раз при первой загрузке. Если правили его уже на
> работающем сервере — перезапуск установки: `cloud-init clean --reboot`.

Если cloud-init по какой-то причине недоступен — используйте Вариант Б ниже.

# Вариант Б. Установка вручную (по шагам)

## 0. Что нужно заранее

- **VPS**: 2 vCPU / **2 ГБ RAM** / 40 ГБ диск, ОС **Ubuntu 24.04** (чистая).
  2 ГБ важны — на 1 ГБ сборка может упасть по памяти (см. шаг 3 про swap).
- **Доступ к серверу**: после создания VPS в панели Timeweb будут **IP-адрес**
  и **пароль root** (или вы зададите свой). Они понадобятся в шаге 1.
- Репозиторий проекта на GitHub (ветка `main` — она «чистая», готова к деплою).

---

## 1. Подключиться к серверу (с Windows)

Откройте **PowerShell** (Пуск → наберите «PowerShell») и подключитесь по SSH.
Подставьте свой IP вместо `<IP>`:

```powershell
ssh root@<IP>
```

- При первом подключении спросит «Are you sure… (yes/no)» — введите `yes`.
- Затем спросит пароль — вставьте пароль root из панели Timeweb
  (при вводе пароль не отображается — это нормально, просто вставьте и Enter).

Если всё ок — приглашение сменится на `root@...:~#`. Дальше все команды
выполняются **на сервере**, в этом окне.

---

## 2. Установить Docker

Docker — это то, что запускает приложение и базу в «контейнерах».
Установка одной командой (вставьте целиком и Enter):

```sh
curl -fsSL https://get.docker.com | sh
```

Подождите 1–2 минуты. Проверьте, что установилось:

```sh
docker --version
docker compose version
```

Обе команды должны напечатать номер версии. Если да — Docker готов.

Git обычно уже есть. Проверьте `git --version`; если «command not found»:

```sh
apt update && apt install -y git
```

---

## 3. (только для 1 ГБ RAM) Добавить swap

Если взяли сервер с **2 ГБ — пропустите этот шаг.**
Если 1 ГБ — создайте файл подкачки, иначе сборка может упасть:

```sh
fallocate -l 2G /swapfile && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

---

## 4. Скачать код проекта

Клонируем ветку `main` (подставьте URL своего репозитория):

```sh
git clone -b main https://github.com/<ваш-аккаунт>/zodchie-dn.git zodchie
cd zodchie
```

Дальше все команды выполняются **из папки `zodchie`** (вы уже в ней после `cd`).

---

## 5. Настроить секреты (`.env.production`)

Это файл с паролями. Его НЕ коммитят в git — создаём прямо на сервере.
Скопируйте шаблон:

```sh
cp .env.production.example .env.production
```

Теперь нужно заполнить три значения. Сгенерируем их по очереди.

### 5.1. SESSION_SECRET (подпись для входа в админку)

Сгенерируйте случайную строку:

```sh
openssl rand -base64 32
```

Скопируйте напечатанную строку — вставите её в файл в шаге 5.4.

### 5.2. ADMIN_PASSWORD_HASH (пароль администратора)

Пароль в базе не хранится — хранится его «хеш». Сгенерируйте хеш для своего
пароля одной командой (Docker уже установлен, ничего ставить не надо).
**Замените `ВАШ_ПАРОЛЬ`** на пароль, которым будете входить в админку
(для теста возьмите простой — латиница и цифры, без пробелов и кавычек):

```sh
docker run --rm node:20-alpine sh -c "npm i -s bcryptjs && node -e \"console.log(require('bcryptjs').hashSync('ВАШ_ПАРОЛЬ',10))\""
```

Напечатает строку из 60 символов, начинающуюся с `$2b$10$…` — это и есть хеш.
Скопируйте его **целиком и как есть** (символы `$` не трогаем, ничего не
экранируем — в этом файле они не требуют обработки).

> Примечание: в проекте есть скрипт `prisma/hash-password.ts`, но он печатает
> хеш с экранированием (`\$`) — это для другого файла (`.env` при локальной
> разработке). Для `.env.production` нужен **сырой** хеш из команды выше.

### 5.3. Пароль базы (по желанию, но лучше сменить)

В шаблоне база защищена паролем `zodchie`. Для теста можно оставить (база
не открыта наружу). Если хотите сменить — поменяйте в **двух местах**:
в `docker-compose.prod.yml` (`POSTGRES_PASSWORD`) и в `DATABASE_URL` внутри
`.env.production` — пароль должен совпадать.

### 5.4. Вписать значения в файл

Откройте файл редактором `nano`:

```sh
nano .env.production
```

Приведите его к такому виду (вставьте свои значения из 5.1 и 5.2):

```
NODE_ENV=production
DATABASE_URL=postgresql://zodchie:zodchie@db:5432/zodchie
SESSION_SECRET=сюда-строку-из-openssl
ADMIN_PASSWORD_HASH=$2b$10$сюда-хеш-из-60-символов
SESSION_COOKIE_INSECURE=true
```

- `SESSION_COOKIE_INSECURE=true` оставляем, пока сайт работает по `http://`
  без сертификата (иначе вход в админку не сработает). Уберём после HTTPS.

Сохранить и выйти из nano: **Ctrl+O**, затем **Enter** (сохранить),
потом **Ctrl+X** (выйти).

---

## 6. Запустить сайт

Собираем образ и запускаем (первый раз — 3–7 минут, идёт сборка):

```sh
docker compose -f docker-compose.prod.yml up -d --build
```

Контейнер сам применит миграции базы и запустит сайт. Проверьте, что оба
контейнера в статусе `running` / `healthy`:

```sh
docker compose -f docker-compose.prod.yml ps
```

---

## 7. Заполнить демо-данными (по желанию)

Чтобы на сайте сразу были категории и товары для показа:

```sh
docker compose -f docker-compose.prod.yml exec app npx prisma db seed
```

Если хотите начать с пустого каталога и наполнять через админку — пропустите
этот шаг.

---

## 8. Открыть порт и зайти на сайт

Откройте порт 3000 в фаерволе (если включён ufw):

```sh
ufw allow OpenSSH
ufw allow 3000
ufw --force enable
```

Откройте в браузере:

- Сайт: `http://<IP-сервера>:3000`
- Админка: `http://<IP-сервера>:3000/admin/login` — пароль тот, что вы
  «захешировали» в шаге 5.2.

> В панели Timeweb может быть свой сетевой фаервол — если сайт не открывается,
> проверьте, что там тоже разрешён входящий порт 3000.

---

## 9. Если что-то не работает — диагностика

Посмотреть логи приложения (последние строки и дальше в реальном времени):

```sh
docker compose -f docker-compose.prod.yml logs -f app
```

(выйти из просмотра логов — **Ctrl+C**)

**Не пускает в админку с верным паролем?** Проверьте, что хеш доехал целым
(должно быть 60 символов, начинается с `$2b$`):

```sh
docker compose -f docker-compose.prod.yml exec app printenv ADMIN_PASSWORD_HASH
```

Если строка короче или без `$2b$` в начале — заново вставьте сырой хеш в
`.env.production` и перезапустите: `docker compose -f docker-compose.prod.yml up -d`.

---

## 10. Управление

```sh
# остановить (данные сохранятся)
docker compose -f docker-compose.prod.yml stop

# запустить снова
docker compose -f docker-compose.prod.yml start

# полностью остановить и удалить контейнеры (тома с базой и фото остаются)
docker compose -f docker-compose.prod.yml down
```

---

## 11. Обновить версию сайта

Когда выйдет новая версия в ветке `main`:

```sh
cd zodchie
git pull
docker compose -f docker-compose.prod.yml up -d --build
```

---

## 12. Бэкапы (данные)

Фото товаров лежат в томе `uploads`, база — в `pgdata`; они переживают
пересборку и перезапуск. Сделать резервные копии:

```sh
# база данных
docker compose -f docker-compose.prod.yml exec db pg_dump -U zodchie zodchie > backup.sql

# загруженные фото
docker run --rm -v zodchie_uploads:/u -v "$PWD":/b alpine tar czf /b/uploads.tgz -C /u .
```

---

## 13. HTTPS и нормальный домен (когда понадобится)

Для теста по IP достаточно `http://…:3000`. Для «боевого» варианта нужен
домен и HTTPS (замок в браузере, безопасный вход в админку). Делается через
реверс-прокси **Caddy** (он сам получает бесплатный сертификат). Когда будете
готовы — скажите домен, добавлю Caddy в `docker-compose.prod.yml`, и тогда
из `.env.production` можно убрать `SESSION_COOKIE_INSECURE`.

---

## Про хранение картинок

Сейчас фото товаров хранятся на диске сервера (том Docker `uploads`) — это
просто, бесплатно и достаточно для одного сервера. Если в будущем понадобится
несколько серверов, CDN или переезд на serverless — переключим загрузку на
S3-совместимое хранилище (Cloudflare R2 / Yandex Object Storage / MinIO),
поменяв только `src/lib/uploads.ts`.
