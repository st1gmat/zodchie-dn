Design direction chosen for the "Зодчие" plumbing-store landing page:
- **Visual style (updated 2026-06-22):** light palette, e-commerce-catalog-style structure inspired by https://www.santehcentr.com/ — light palette with a **soft blue / sky-blue pastel** scheme (changed 2026-06-23 from the earlier warm brass/copper accent — user wanted pastel сине-голубой that doesn't strain the eyes): the `:root` vars in `src/app/globals.css` are now `--background:#f5f9fd` (blue-tinted off-white, not stark white), `--surface:#eaf2fa`, `--foreground:#1e2a38` (soft dark navy, not black), `--muted:#5d7387`, `--border:#d3e1ee`, accent ramp `#5e9ed1`/`#4585bd`/`#346a9c`, plus `--card:#fbfdff` (a near-white, barely-tinted surface used uniformly for product cards — the user disliked both the bluish `--background` and pure white there; pure white "strained the eyes").
- **Product characteristics + category templates (2026-06-25, santehcentr-style):** `Product` gained `code` (код товара), `brand` (бренд), `article` (артикул) string fields, shown as a compact meta block in the **top panel** of the product page (next to title/price/stock). Top-right panel order (settled per user): title, price, "в наличии", then the **meta as a vertical list** (код товара / бренд / артикул, each on its own line), then **description**, then the "Заказать по телефону" CTA + note in their normal flow (an `mt-auto` bottom-pinned version was tried and rejected — "looked bad"). The `ProductAttribute` **characteristics table sits below the panel**, full width, and flows into **two columns when there are >6 attributes** (`sm:grid-cols-2`). Demo products are seeded with ~9 attributes each (`specsByIcon` + a shared `commonSpecs` list, de-duped by name) plus a generic description, so the 2-column layout and the description block are populated. New `CategoryAttribute` model = a per-category **template** of characteristic *names* (e.g. Артикул, Серия, Цвет) so the admin doesn't re-type them per product; seeded on top categories from `specsByIcon` names. Admin: `ProductForm` has code/brand/article inputs; the product edit page has a client `ProductAttributesEditor` (add/remove/edit rows, "Добавить из шаблона категории" pulls the template names — uses the product's own category template, falling back to the parent's since products live in subcategories); the category edit page has a `CategoryTemplateEditor`. Both editors call **array-argument server actions** (`saveProductAttributes(productId, rows)`, `saveCategoryTemplate(categoryId, names)` in `actions.ts`) directly from the client via `useTransition` — each does a `deleteMany`+`createMany` transaction. Reminder: products are in subcategories, so a product's public URL uses its **subcategory** slug, not the top category's.
- **Header/footer/social (2026-06-25):** the top utility bar (above the main header) now has Telegram + Instagram icon links (`src/components/icons.tsx`, shared with the footer; URLs are placeholders in `siteConfig.social`). Header nav gained **«Главная»** alongside «Каталог». New **`Footer.tsx`** (client, hides on `/admin` via `usePathname` like the Header; rendered in the root `layout.tsx` after `<main>`): 4 columns — brand+social, Разделы, Контакты, Реквизиты — plus a bottom legal bar. Kept the footer **static (no DB fetch in the layout)** on purpose so `/` and `/catalog` stay statically prerendered. `siteConfig` gained `email` and `social.{telegram,instagram}`.
- **Search now matches characteristics too (2026-06-25):** `/api/search` product query is an `OR` over title, **brand, article, code, and `attributes.some.{value,name}`** — so typing e.g. «латунь» or «SanLux» finds products by their specs/brand, not just the name.
- **Header search (2026-06-25):** `SearchBar.tsx` (client) sits in the header next to the logo (and inside the mobile menu). It debounces (130ms) a fetch to the `/api/search` route handler (`src/app/api/search/route.ts`), which runs case-insensitive `contains` queries on product + category titles (up to 6 products / 5 categories). The dropdown shows thumbnails, prices, parent-category context, bold-highlights the matched substring, and Enter jumps to the first hit. Animations are plain `@keyframes` in `globals.css` (`dropdown-in`, and `result-in` with a per-item `animationDelay` stagger). **Turbopack-dev gotcha (confirmed worse than thought):** edits to `globals.css` (`:root` vars, `@keyframes`, …) aren't picked up live, AND a plain `npm run dev` restart can still serve a stale globals chunk from the Turbopack cache — the reliable fix is **stop dev → delete `.next/dev` → restart**. Component/className edits hot-reload fine. All components read these tokens, so the whole site re-themes from this one block; the hero gradient also uses `var(--background)`. Header has: top utility bar (address/hours/phone), main row (logo, nav, favorites+cart icon links, "Позвонить" button) — no category row in the header. Superseded the earlier "dark premium, deliberately not like competitors" direction from 2026-06-20 — user changed their mind after seeing santehcentr.com and explicitly asked to adopt its structure + light colors.
- **Category nav placement (corrected 2026-06-22):** on santehcentr.com the category list is a narrow left **sidebar** (`<aside class="sidebar col-lg-2">`, confirmed by reading the raw HTML — not a full-width block as an earlier WebFetch summary wrongly suggested) sitting next to the hero/carousel (`col-lg-10`), not inside or under the header. We matched this: `src/components/CatalogSidebar.tsx` (vertical category list) renders inside `Hero.tsx`. Visibility (updated 2026-06-23): the catalog used to be `hidden lg:block` (vanished below 1024px — the user couldn't see it and asked to keep it on the left); now it's **always rendered** — the hero grid is 2-col from `md` (`md:grid-cols-[190px_minmax(0,1fr)]` — sidebar narrowed from 240→190px on 2026-06-23 per a user mockup that wanted a slimmer catalog + a wider heading block), and on phones it stacks **below** the hero text via `order` (text `order-1`, sidebar `order-2 md:order-1`) so the heading stays first. Layout (settled 2026-06-23 after several iterations): the hero inner container is **centered** `mx-auto max-w-7xl px-6` (aligns the catalog with the header logo; full-bleed photo still spans the whole section behind it). A left-aligned full-width version was tried but the user disliked that it hugged the left edge when zoomed out — centered keeps the content balanced on wide/zoomed-out viewports and falls back to full-width on narrow ones. The grid is `md:grid-cols-[320px_minmax(0,1fr)]` (catalog 320px) + `gap-8`, and the heading block has `md:pl-8` so it isn't glued to the catalog. The catalog fills its 320px track normally (no negative-margin pull — see the cautionary note below). **This replaced an earlier hacky approach** that kept the container centered and pulled the catalog left with a negative `calc()` margin — avoid going back to that: a `%`-based margin over-pulled it off-screen (grid-item percentage margins don't resolve against the grid container width), and even the px+vw version (`calc(616px - 50vw)`) had a nasty side effect — a grid item defaults to `justify-self: stretch`, so a *negative* margin grows its width, making the catalog stretch wider on wider monitors. The full-width left-aligned layout sidesteps all of it.
- **Hero text contrast (fixed 2026-06-23):** `hero-bathroom.png` is **light on the left, dark grey stone wall on the right**. The hero heading/description are `text-foreground`/`text-muted` (dark); once the catalog-left layout pushed the text rightward, the right end of the heading landed over the dark wall and disappeared (user reported "white/invisible text"). Fix: the overlay gradient is an inline style `linear-gradient(to right, var(--background) 0%, var(--background) 58%, transparent 92%)` (solid light on the left, photo reveals on the right) and the heading block is `max-w-2xl`. These two numbers (gradient solid-stop % and heading max-width) are the knobs that trade off "wider heading" vs "more visible photo" vs "contrast" — tune them together if the user asks for a wider/narrower heading. If the hero photo is ever swapped, re-check this — dark text over a dark-right photo is the failure mode. **Hero relaid out 2026-06-23** (user found the old version awkward): the old hero was a full-bleed background photo with a *floating* collapsible sidebar card and the heading `justify-center`d behind a huge `pr-56` (it "floated" with no anchor, and the photo's right half was wasted padding). Now `Hero.tsx` keeps the **full-bleed background photo** (full width, `fill` + left→right `from-background via-background/85 to-transparent` gradient — user explicitly wanted the full-width background, NOT a bordered/contained banner card — a contained-card version was tried and rejected), and overlays a contained `max-w-7xl` CSS grid `lg:grid-cols-[260px_minmax(0,1fr)]` on top: left = the catalog list (now a **persistent, non-collapsible server component** — dropped the `useState` toggle), right = left-aligned eyebrow/h1/description/CTAs (`max-w-2xl`, `items-start`). The sidebar card is now anchored to the page container's left edge instead of floating. The homepage section below the hero is now `ProductShowcase` (added 2026-06-23): a client-side carousel of **random** products — 8 per page (4×2 grid), **max 3 pages** (24 products), prev/next arrows + dot pagination, translateX paging. Data from `getShowcaseProducts()` in `catalog.ts` (Fisher–Yates shuffle, frozen per build since `/` is static). The old `CategoryGrid` icon-tiles section was **removed** from the homepage and its component file **deleted** (user asked to drop it). Categories are still reachable via the hero sidebar + the `/catalog` index + a "Весь каталог →" link in the showcase. **Card image flipping (2026-06-23):** `ProductCard` is now a client component — `ProductCardView` carries `images: string[]` (was single `imageUrl`; placeholder when none), and on a card with ≥2 photos, moving the cursor across horizontal hover-zones flips between them (Wildberries-style) with thin position dashes on top. Every product is now seeded with **2 demo photos** (`/images/demo-bathroom.avif` + `/images/product-placeholder.png`) so the gallery and card-flip are populated out of the box. The seed only adds them when a product has **0** images (`imageCount === 0` guard), so re-seeding never wipes photos uploaded via the admin. (Note: the product detail page was never actually broken when the user reported "can't find images" — it was just showing the grey placeholder because no real photos existed yet.) Don't trust a single WebFetch text summary for pixel/layout-level structure questions again — grep the raw HTML (Bootstrap col-* classes, aside/sidebar markup) when the user pushes back on a layout claim.
- **Cart/favorites icons:** currently decorative-only (link to `#catalog`, `title` tooltip "появится позже") — no real cart/wishlist logic exists yet; don't treat the icons as a sign that cart functionality is implemented.
- **Content:** placeholder copy/icons for now — no real product photos or logo yet. Don't fabricate fake stats/numbers (e.g. "10 years on market", review counts); generic marketing claims (delivery, consultation) are fine, factual-sounding numbers are not.
- **Build approach:** stepwise/iterative, not one giant page at once. First slice built (2026-06-20): `Header`, `Hero`, `CategoryGrid` (8 placeholder categories) in `src/components/`, config in `src/lib/site.ts`. Restyled to light/santehcentr-structure on 2026-06-22 (same components, no new sections).
- **DB layer wired (2026-06-22):** moved categories off the static `site.ts` array onto Postgres via Prisma 7. Schema: `Category` (self-relation parent/children + `icon` string keyed to `CategoryIcon.tsx` map like "faucets"/"toilets"), `Product`, `ProductImage` in `prisma/schema.prisma`. Front-end now reads from DB: `src/lib/categories.ts` `getCategories()` (top-level, ordered) → `page.tsx` is an async server component that passes `categories` into `Hero`→`CatalogSidebar` and `CategoryGrid` as props. The static `categories`/`Category` export was removed from `site.ts` (siteConfig contacts/legal stay static there). Gotchas for this stack: Postgres runs in Docker on **host port 5433** (5432 was taken by another project's `ms_pg_sql` container) — see `docker-compose.yml`/`.env`; Prisma 7's `prisma-client` generator outputs TS to `src/generated/prisma`, import `PrismaClient` from `@/generated/prisma/client` (no index); it **requires a driver adapter** — `new PrismaClient({ adapter: new PrismaPg({ connectionString }) })` (`@prisma/adapter-pg` installed), bare `new PrismaClient()` throws; seed runs via `npx tsx` (config `seed`) and needs `import "dotenv/config"`; re-run `prisma generate` after schema field changes or the client validation lags. `/` currently prerenders **static** at build (categories baked in) — add `revalidate`/`force-dynamic` once the admin panel can edit categories at runtime.
- **Public catalog pages built (2026-06-22):** `/catalog` (category index), `/catalog/[slug]` (category + product grid), `/catalog/[slug]/[product]` (product detail). Data accessors in `src/lib/catalog.ts` (`getCategoryDetail`, `getProductDetail` — the latter validates the product actually belongs to `categorySlug`, else `notFound()`, to avoid duplicate URLs); price formatting in `src/lib/format.ts` (whole rubles, `ru-RU` nbsp). Components: `ProductCard.tsx`, `Breadcrumbs.tsx`. `Header` was moved out of `page.tsx` into the root `layout.tsx` so it renders on every route (and `<main>` lives there now); home `page.tsx` returns just `<Hero>`+`<CategoryGrid>`. All internal nav switched from `#catalog` hash anchors to `next/link` real routes (Header logo/nav/icons, Hero CTA, `CategoryGrid` cards, `CatalogSidebar`). Seeded **32 demo products** (4 per category, a couple `inStock:false`) — clearly placeholder nomenclature until real/1C data; products have no photos so cards fall back to `/images/product-placeholder.png` (WooCommerce-style no-image placeholder the user supplied, in `public/images/`). The `[slug]` and `[slug]/[product]` routes are **dynamic (SSR on demand)** — no `generateStaticParams` — so they reflect DB edits immediately (unlike static `/` and `/catalog`). Next 16 dynamic-route notes: `params` is a Promise (`await params`), type pages with the `PageProps<'/route'>` global helper, `notFound()` from `next/navigation`.
- **Subcategories (2026-06-23):** the `Category` self-relation (`parentId`/`children`) is now used. Seed adds 4 children per top category (`subsByIcon`; child slug = `${parentSlug}-${slugify(title)}` for uniqueness). The 4 demo products per category are **assigned to its subcategories** (round-robin, 1 per sub — so no product sits directly on a top-level category), making subcategory pages non-empty while parent pages still aggregate everything beneath them. `getCategories()` returns top-level **with `children`**; `CatalogSidebar` shows a **CSS-only hover flyout** (no JS) — `group/cat` + `group-hover/cat:` opacity/translate/visibility transitions, panel positioned `absolute left-full` with a `pl-2` hover bridge; the sidebar card dropped `overflow-hidden` (would clip the flyout) and the `<aside>` got `z-30` so the flyout sits above the hero text. The hero `<section>` also had to drop `overflow-hidden` and gain `z-20` (2026-06-23): a flyout from a low category would otherwise be clipped at the section's bottom edge; now it overflows and paints on top of the section below. Removing the section's `overflow-hidden` is safe — the full-bleed `<Image fill>` and gradient are `inset-0` (already bounded) and there's no negative-margin content anymore. A category page (`getCategoryDetail`) aggregates products of the category **plus its children**, shows child chips, and breadcrumbs include the parent. Important: products carry their **own** `categorySlug` in `ProductCardView` so links from an aggregated parent page resolve (else `getProductDetail`'s category-match check 404s). Admin: `CategoryForm` has a parent `<select>` (top-level only, can't self-parent), `updateCategory` guards self-parent, deleting a parent sets children to top-level (relation `onDelete: SetNull`) while its own products cascade-delete; the category list shows the tree indented; product category `<select>` is hierarchical via `getCategoryOptions()` (children prefixed "— "). Only **one** nesting level is rendered in the menu. (Note: the user had already created a stray top category "Канделябры" via the admin earlier — real data, left as-is.)
- **Admin panel built (2026-06-22):** single-admin auth — no user table, password compared (`bcryptjs`) against `ADMIN_PASSWORD_HASH`, session is a `jose` HS256 JWT in an httpOnly cookie `admin_session` signed with `SESSION_SECRET`. Helpers in `src/lib/auth.ts` (`verifyPassword`, `createSession`, `destroySession`, `isAuthenticated`, `requireAdmin`); `prisma/hash-password.ts` prints a hash (`npx tsx prisma/hash-password.ts <pwd>`). **Local dev admin password is `zodchie-admin`** (hash sits in gitignored `.env`). **Gotcha (cost a debugging round):** Next's env loader (`@next/env`) expands `$...` in `.env` values, so a raw bcrypt hash like `$2b$10$...` gets mangled (only the tail survives) and login silently fails — every `$` in `ADMIN_PASSWORD_HASH` must be escaped as `\$` in `.env`. `prisma/hash-password.ts` now prints a paste-ready escaped line. A bare `dotenv/config` test does NOT reproduce this (no expansion), so test env-dependent auth the way Next loads it. Pages: `/admin/login` (public) + a `(panel)` route group whose `layout.tsx` calls `requireAdmin()` and renders admin chrome — dashboard `/admin`, `/admin/categories` (+`[id]` edit, inline create), `/admin/products` (+`new`, `[id]` edit). Mutations are Server Actions in `src/app/admin/actions.ts` (`"use server"`, every action re-checks `requireAdmin()`, validates with `zod`, then `revalidatePath("/", "layout")` so the static storefront refreshes). No middleware/proxy used (Next 16 renamed middleware→**proxy**; guard via layout + per-action check instead). The storefront `Header` hides itself on `/admin` via `usePathname()` (avoided a route-group refactor of the public pages, which was blocked by a Windows file lock from the running dev server). Slug is set on create and **left unchanged on edit** (keeps URLs stable).
- **Product image uploads (2026-06-22):** managed on the product edit page (`ProductImagesManager.tsx`) — multi-file upload, delete, and "сделать главной" (renumbers `ProductImage.order`; order 0 = primary, the one shown on cards). Files stored on **local disk** in `public/uploads/` (gitignored, served at `/uploads/...`); `src/lib/uploads.ts` validates type (jpeg/png/webp/avif) + 8MB cap and writes with a uuid filename; delete also unlinks the file. Actions (`addProductImages`/`deleteProductImage`/`makeProductImagePrimary`) live in `src/app/admin/actions.ts`. Server Actions cap request bodies at 1MB by default, so `next.config.ts` raises `experimental.serverActions.bodySizeLimit` to `8mb`. The public gallery picks up uploaded images automatically (falls back to placeholder when none). **Deployment caveat:** local-disk storage assumes a persistent writable filesystem (self-hosted/Docker/VPS) — it won't work on read-only/serverless (e.g. Vercel); switch to object storage (S3/MinIO) there. NOT done in admin yet: editing `ProductAttribute` specs (still seed-only), login rate-limiting, multiple admins. The fuller storefront (cart, comparison) like santehcentr.com is still deferred.
- **Still to build:** advantages/"why us" section, "about company" (using real legal info from [[project-business-info]]), contacts section with map, footer with legal/INN details. Plus the admin panel above. Product detail page now has a `ProductGallery` (client, thumbnails only when >1 image) and a "Характеристики" spec list from the new `ProductAttribute` model (relational, mirrors `ProductImage`; seeded ~4 demo specs per category via `specsByIcon`, reset+recreated each seed for idempotency). Still no cart. 1C nomenclature import still undecided per README/CLAUDE.md.

**Why:** user explicitly rejected a straight clone of nautilusdn.com on 2026-06-20, but on 2026-06-22 asked to adopt santehcentr.com's structure and light colors instead — direction is now light, not dark. Always confirmed scope/details via AskUserQuestion before implementing rather than assuming.
**How to apply:** Treat the light palette + santehcentr-style header structure as current truth, not the older dark-premium note (still useful for icon-style/whitespace conventions, just not colors). Continue checking in with the user before adding new sections or scope (catalog, admin panel).
